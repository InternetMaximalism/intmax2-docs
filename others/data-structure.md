# Data Structure

## Version

**Version Number**: v0.2.0

**Update Date**: 16.06.2024

## Purpose

The purpose of this document is to ensure data consistency and efficiency in system development and operation.

## Data Types

### Address

A 20-byte string used to identify accounts and contracts on Ethereum.

### Elements of the Goldilocks Finite Field

Integers ranging from $0$ to $2^{64} - 2^{32} + 1$.

### Hash

The output of a one-way hash function. The following hash functions are used:

- Goldilocks Poseidon
    - Function: `Poseidon(inputs)`
        - Input:
            - `inputs`: An array of arbitrary length containing elements from the Goldilocks field.
            - **NOTE**: If the input is provided as a byte array, it should be divided into 4-byte segments and converted to elements of the Goldilocks finite field.
        - Output:
            - A value consisting of four elements from the Goldilocks finite field.
- Keccak256
    - Function: `Keccak256(inputs)`
        - Input:
            - `inputs`: A byte array of arbitrary length.
        - Output:
            - A 32-byte array.
- Sha256
    - Function: `Sha256(inputs)`
        - Input:
            - `inputs`: A byte array of arbitrary length.
        - Output:
            - A 32-byte array.

### Merkle Tree

A data structure used to efficiently verify data integrity, consisting of leaf nodes, internal nodes, and a root node.

- Function: `rootHash = MerkleRoot(leaves)`
    - Input:
        - `leaves`: An array of hash values.
    - Output:
        - `rootHash`

Regarding storage in the database:

- As long as all leaves are stored, the tree can be reconstructed.
- For convenience, it is also advisable to store the root hash.
- To further optimize the reconstruction of Merkle proofs for past root hashes, cache the correspondence from internal nodes to their child nodes.

### Plonky2 Proof

A component of the ZKP system, consisting of the following two elements:

- `proof` (string)
- `public inputs` (string)

## Data Items

### Public Key

A public key used for transaction signature verification in INTMAX.

- The private key is randomly selected from the scalar field of the BN254 elliptic curve, and the public key is generated by multiplying the generator by this value.
    - It can be derived from the Ethereum account private key.
- The default value is a point on the BN254 elliptic curve where the x-coordinate is 1.
    - **NOTE**: There is no point on the BN254 elliptic curve where the x-coordinate is 0.

### Token Index

Sequential token addresses assigned in the order they are first deposited into the INTMAX network.

The liquidity contract stores this mapping. The contract has a mapping for `token_index` ⇒ (token_address, token_id).

- Size: 4 bytes
- Data Type: Integer

### Generic Address

A public key or Ethereum address.

- `addressType`: `ETHEREUM`, `INTMAX`
- `address`: A 20-byte address (`ETHEREUM`) or a 32-byte public key (`INTMAX`)

### Transfer Data

```go
type TransferData struct {
	recipient struct{ addressType, address string }
	amount    big.Int
	salt      big.Int
}
```

Example (JSON):

```json
{
    "recipient": {
	    "addressType": "INTMAX",
	    "address": "0xabc...def"
	  },
    "amount": "123",
    "salt": "0xabc...def"
}
```

- `recipient`: Generic address
    - **NOTE**: The generic address is only used in transfer data.
- `amount`: The amount of tokens to be transferred (32 bytes).
- `salt`: Four values randomly selected from the Goldilocks finite field (32 bytes).
    - Salt is included to ensure that even if the same transfer is executed twice, the second transfer is not mistakenly identified as a duplicate.

**NOTE**: The token index is not specified here but is hashed with the root of the transfer tree.

**Transfer Data Hash**

- `transferHash = Poseidon(recipient, amount, salt)`
    - `recipient`: A 32-byte string created from the recipient of the transfer data.
        - If `addressType` is `ETHEREUM`, pad the 20-byte `address` with 12 bytes of 0 at the beginning to make it a 32-byte string.
        - If `addressType` is `INTMAX`, the first 2 bits of the 32-byte `address` are always 0, so change the first bit to 1 to distinguish it from an Ethereum address.
    - `amount`: Consider the amount of the transfer data as a 32-byte string.
    - `salt`: Consider the salt of the transfer data as a 32-byte string.

### Fee Transfer Data

Payment details to the Block Builder.

- `recipient`: The Block Builder's public key (32 bytes).
- `tokenIndex`: The address of the token to be transferred on INTMAX (4 bytes).
- `amount`: The amount of tokens to be transferred (32 bytes).
- `salt`: Four values randomly selected from the Goldilocks finite field (32 bytes).
    - The significance of salt is the same as in transfer data.

Example (JSON):

```json
{
    "recipient": "0xabc...def",
    "tokenIndex": 0,
    "amount": "123",
    "salt": "0xabc...def"
}
```

**Fee Transfer Data Hash**

- `feeTransferHash = Poseidon(recipient, tokenIndex, amount, salt)`
    - `recipient`: A 32-byte string created from the recipient of the fee transfer data.
        - If `addressType` is `ETHEREUM`, pad the 20-byte `address` with 12 bytes of 0 at the beginning to make it a 32-byte string.
        - If `addressType` is `INTMAX`, the first 2 bits of the 32-byte `address` are always 0, so change the first bit to 1 to distinguish it from an Ethereum address.
    - `tokenIndex`: Consider the token index of the fee transfer data as a 4-byte string.
    - `amount`: Consider the amount of the fee transfer data as a 32-byte string.
    - `salt`: Consider the salt of the fee transfer data as a 32-byte string.

**NOTE**: Here, `recipient` refers to the Block Builder's address, not the recipient of the transferData.

### Transfer Tree

A Poseidon Merkle tree containing up to 128 transfer data items of the same token type.

- Leaf Format:
    - `transferLeafHash = Poseidon(recipient, amount, salt)`
        - `recipient`: Split the 32-byte string into eight 4-byte segments.
        - `amount`: Split the 32-byte string into eight 4-byte segments.
        - `salt`: Split the 32-byte string into eight 4-byte segments.
- `transferMerkleRoot = MerkleRoot(transferLeafHashes[0], …, transferLeafHashes[127])`
    - `transferLeafHashes`: Up to 128 transfer data hashes created using the above method.
    - **NOTE**: Different types of tokens are not mixed to significantly reduce the size of the Plonky2 circuit.

### Transfers Hash

Hash of transfer information.

`transfersHash = Poseidon(transferTreeRoot, tokenIndex)`

- `transferTreeRoot`: The root hash of the transfer tree (32 bytes).
- `tokenIndex`: The address of the token on INTMAX (4 bytes).

### Nonce

The number of transfer transactions executed with the INTMAX account. Managed by the user.

### Transaction

![2024-06-14 23.01.40.png](/assets/2024-06-14_23.01.40.png)

Transfer information including fee data. A transaction contains the following data:

- `feeTransferHash`: The hash of the fee transfer data (32 bytes).
- `transfersHash`: The hash value calculated from the transfer tree and token index defined above.
- `nonce`: The number of transfer transactions executed with the INTMAX account.
- **NOTE**: It is crucial to maintain the order of transactions within the tx tree. Swapping positions between users with different transaction amounts can cause issues.
    - Verified by ZKP:
        - A transaction must be at a specific index within the tx tree.
        - The index is determined by the sender list.
        - If swapping occurs:
            - User A signing a transaction might send 1 unit instead of 1 million units.
            - User B might attempt to send 1 million units despite having only 1 unit, making the transaction invalid and locking the account until sufficient funds are added.
            - Users should avoid signing incorrect transactions.
- The sender's public key is also provided to the Block Builder.

**Transaction Hash**

- `txHash = Poseidon(feeTransferHash, transfersHash, nonce)`

Example (JSON):

```json
{
    "feeTransferHash": "0x123...456",
    "transfersHash": "0xabc...def",
    "nonce": 1
}
```

### Tx Tree

![2024-06-14 21.33.44.png](/assets/2024-06-14_21.33.44.png)

A Poseidon Merkle tree containing transactions from different senders as leaves. Each tree can have up to 128 leaves.

- `txRoot`: The root hash of the tx tree.

### Sender Type

- 1: PUBLIC_KEY
    - The first transfer by the public key.
- 2: ACCOUNT_ID
    - Transfers from the second transaction onwards by the public key.

### Sender List

The public key (254 bits) or account ID (5 bytes) of each sender.

- For the first transaction by a public key, the public key must be included in calldata to prevent rogue key attacks.
- A block must contain transactions from either PUBLIC_KEY or ACCOUNT_ID senders, not both.
- Order by descending value (use the default value if less than 128 senders).

### Block

To construct a block, the types of transactions within it must be consistent. Thus, a block must not mix transactions specified by public keys with those specified by account IDs, nor should it mix transfers and withdrawals.

![2024-06-14 21.56.50.png](/assets/2024-06-14_21.56.50.png)

- `prevBlockHash`: The previous block hash.
- `blockNumber`: The block number, which is the latest block number in the rollup contract plus 1.
- `depositRoot`: The deposit root at the time of block posting (written in the rollup contract).
- `contentHash`: The hash value that the Block Builder must provide to the rollup contract when posting a new block.
    - `contentHash = Keccak(txRoot, signatureHash, senderType, senderFlags, accountIdsHash, senderPublicKeysHash)`
        - `senderType`: PUBLIC_KEY or ACCOUNT_ID
        - `senderPublicKeysHash = Keccak(senderPublicKeys)`
            - Even when provided with account IDs, the senders hash uses public keys.
            - The contract does not verify that `senderPublicKeysHash` is calculated from the sender list.
                - **NOTE**: Assigning account IDs to submitted public keys requires ZKP circuits.
        - `senderAccountIdsHash`:
            - If `senderType` is `ACCOUNT_ID`:
                - `senderAccountIdsHash = Keccak(senderAccountIds)`
                    - Padless 5-byte values.
                    - Calculated in the contract.
            - If `senderType` is `PUBLIC_KEY`:
                - `senderAccountIdsHash = 0`
        - `aggregatedSignature`: The aggregated signature (64 bytes).
            - Uncompressed format.
            - Use BDN18 ([https://eprint.iacr.org/2018/483](https://eprint.iacr.org/2018/483)). Hash all sender public keys, add the own public key to the hash, and use the result as a weight for multiplying with the private key. Sum the resulting public keys for all senders.
        - `aggregatorPublicKey`: The aggregated public key (64 bytes).
            - Uncompressed format.
        - `messagePoint`: The tx tree root hash converted into a format that can be signed with BLS (128 bytes).
            - Uncompressed format.
            - `messagePoint = HashToG2(txRoot)`
        - `signatureHash`: The hash of signature-related data.
            - `signatureHash = Keccak(aggregatedSignature, aggregatedPublicKey, messagepoint)`
        - `senderFlags`: A 16-byte value representing whether each sender has signed, ordered the same as the sender list.
            - Each bit represents whether the sender signed:
                - **Signed (1)**: The bit is set to 1.
                - **Not Signed (0)**: The bit is set to 0.
                - **Order**: Bits correspond to senders in the order of the sender list.
                
                **NOTE**: If no one signs, the aggregated signature cannot be verified within the ZKP.
                
            - For the initial transaction, the sender must return a signature to be assigned a new account ID.
                - Without a returned signature, proof of ownership (PoP) for the public key cannot be made.

**Block Hash**

`block_hash = Keccak(prev_block_hash, block_number, content_hash, deposit_root)`

- Store the block hash each time:
    - `mapping(bytes32 => uint256) blockHashes`
        - `block hash` ⇒ `block number`
    - Output the event log `BlockPosted(prevBlockHash, blockNumber, contentHash, depositRoot)`.
- The genesis block sets `prev_block_hash`, `block_number`, `content_hash`, and `deposit_root` to 0.

### PoW Nonce

`powNonce` is a 32-byte value calculated from the transaction hash.

- Iterate the hash calculation until `powNonce` is found such that the first `n` digits of `powResponse = Sha256(transferBatchHash, powNonce)` are 0.
    - The output value `powResponse` does not affect block values or ZKP inputs.
- The `n` value is defined as `DIFFICULTY` for each builder node. The larger the `n` value, the more computational resources the user requires.
- **NOTE**: This mechanism addresses the issue where a builder cannot receive fees from users who do not sign after executing a transfer. The computation load mitigates spam from users.

`powNonce` satisfies the **PoW condition of difficulty `DIFFICULTY`** when the first `n` digits of `powResponse` are 0.

### Merged Transfer Nullifier Tree

An incremental Merkle tree with transfer hashes as leaves. Users individually calculate and reflect these to ensure that transfer data is not received twice.

- Calculated with Poseidon hash.

### Account Tree

An indexed Merkle tree where the index is the account ID and the leaf is a pair of public key and last seen block number. Account IDs start from 1. Hash uses Goldilocks Poseidon.

- If the same public key corresponds to two different account IDs, the balance proof is calculated for each, and the transfers can be merged to receive the same amount twice. To prevent this, the key for the account tree is the public key.

![2024-06-11 17.01.01.png](/assets/2024-06-11_17.01.01.png)

### Block Hash Tree

An incremental Merkle tree where the index is the block number and the leaf is the block info. Block numbers start from 1. Hash uses Goldilocks Poseidon.

- Leaf: block info (block hash, tx root, valid sender public keys)
- `valid sender public keys`: For each sender, the value is retained if they returned a signature; otherwise, it is filled with 0.

![2024-06-11 17.01.15.png](/assets/2024-06-11_17.01.15.png)

### Deposit Tree

An incremental Merkle tree where the index is the deposit index, and the leaf is the deposit request. The deposit index starts from 1. Hash uses Keccak256.

![2024-06-11 17.08.29.png](/assets/2024-06-11_17.08.29.png)

### Withdraw Tree

An incremental Merkle tree where the index is the withdraw index, and the leaf is the withdraw request. The withdraw index starts from 0. Hash uses Keccak256.

![2024-06-11 17.10.06.png](/assets/2024-06-11_17.10.06.png)

### Balance Data

A mapping that correlates token addresses (token index) with their balances on INTMAX. Balances are represented as integers without considering decimals. For example, 0.1 ETH would be represented as 10^17. Balances can be negative.

Example (JSON):

If the user has 300,000,000,000,000 tokens of address 0 and 50,000 tokens of address 2:

```json
{
  "0": "300000000000000",
  "2": "50000"
}
```

### Asset Tree

A Merkle tree constructed with each key-value pair of balance data as a leaf.

Leaf content:

- `is_negative` (bool): Whether the balance is negative.
- `balance` (32 bytes): The absolute value of the balance.

### Block Validity Proof

Data that demonstrates the correctness of block hash transitions. Provided as a Plonky2 proof.

- Public inputs:
    - `last_seen_block_number`: The latest block number reflected.
        - 4 bytes
    - `block_hash`: The block hash corresponding to the last seen block number.
        - 32 bytes
    - `block_hash_tree_root`: The root hash of the block hash tree.
        - 32 bytes
    - `prev_account_tree_root`: The root hash of the account tree.
        - 32 bytes
    - `next_account_id`: The next account ID to be assigned.
        - 5 bytes
    - `tx_root`.
    - The root hash of the latest sender list as a Poseidon Merkle tree.

### Enough Balance Proof

Data that proves the balance is non-negative without revealing the balance itself. Provided as a Plonky2 proof.

- Public inputs:
    - `public_key`: The user's public key.
    - `asset_tree_root`: The root hash of the asset tree (32 bytes).
    - `last_tx_hash`: The hash of the last reflected transfer transaction.
    - `last_block_number`: The block number containing the last reflected transfer.
    - `nullifier_tree_root`: The root of the merged tx nullifier tree (32 bytes).
    - `block_hash`: The block hash of the block validity proof (32 bytes).
    - `nonce`: The number of transactions executed by the user with that public key.
- Update Rules:
    - Send (including withdrawal):
        - Private inputs:
            - Previous enough balance proof (140 KB).
            - Block validity proof (140 KB).
            - Transaction:
                - `fee transfer hash`: The hash of the fee transfer data (32 bytes).
                - `transfer tree root`: The root hash of the transfer tree (32 bytes).
                - `token index` (4 bytes).
                - `nonce` (4 bytes).
                - Transfer data (~10 KB = 81 bytes * up to 128 items):
                    - `recipient` (32 bytes).
                    - `amount` (32 bytes).
                    - `salt` (8 bytes).
            - Fee transfer data:
                - `recipient` (32 bytes).
                - `token index` (4 bytes).
                - `amount` (32 bytes).
                - `salt` (8 bytes).
            - `sender_index`: The index within the sender list.
        - Constraints:
        **NOTE**: Denote old [new] public inputs as `old_pi` [`new_pi`].
            - `new_pi.public_key === old_pi.public_key`.
            - Hash the transaction using Poseidon to get `tx_hash`.
            - `tx_hash` is included in the block info tree of the block validity proof.
            - `sender_list[sender_index] === old_pi.public_key`.
            - The total amount sent within the transaction is `total_sent_amount`.
            - `new_pi.asset_tree_leaves[token_index] === old_pi.asset_tree_leaves[token_index] - total_sent_amount`.
            - Hash the fee transfer data using Poseidon to get `fee_transfer_hash`.
            - `new_pi.asset_tree_leaves[fee_transfer_data.token_index] === old_pi.asset_tree_leaves[fee_transfer_data.token_index] - total_sent_amount`.
            - If `new_pi.asset_tree_leaves[token_index]` is negative:
                - `new_pi.last_tx_hash === old_pi.last_tx_hash`.
            - If `new_pi.asset_tree_leaves[token_index]` is non-negative:
                - `new_pi.last_tx_hash === tx_hash`.
            - `new_pi.last_block_number === old_pi.last_block_number`.
            - The deposit tree root of the block validity proof is `calcMerkleRoot(deposit_hash, deposit_merkle_proof.index, deposit_merkle_proof.siblings)`.
            - `new_pi.nonce === old_pi.nonce + 1`.
            - `new_pi.nullifier_tree_root === old_pi.nullifier_tree_root`.
            - Calculate `new_pi.block_hash` from `old_pi.block_hash`.
    - Receive:
        - Private inputs:
            - Previous enough balance proof (140 KB).
            - Block validity proof (140 KB).
            - Sender's balance proof (140 KB).
            - Transaction:
                - `fee transfer hash`: The hash of the fee transfer data (32 bytes).
                - `transfer tree root`: The root hash of the transfer tree (32 bytes).
                - `token index` (4 bytes).
                - `nonce` (4 bytes).
                - Transfer data (~10 KB = 81 bytes * up to 128 items):
                    - `recipient` (32 bytes).
                    - `amount` (32 bytes).
                    - `salt` (8 bytes).
            - Transfer data (68 bytes):
                - `recipient` (32 bytes).
                - `amount` (32 bytes).
                - `salt` (32 bytes).
        - Constraints:
        **NOTE**: Denote old [new] public inputs as `old_pi` [`new_pi`].
            - `new_pi.public_key === old_pi.public_key`.
            - `recipient === old_pi.public_key`.
            - `new_pi.asset_tree_leaves[token_index] === old_pi.asset_tree_leaves[token_index] + transfer_data.amount`.
            - Hash the fee transfer data using Poseidon to get `fee_transfer_hash`.
            - `new_pi.last_tx_hash === old_pi.last_tx_hash`.
            - `new_pi.last_block_number === old_pi.last_block_number`.
            - `transfer_hash = Hash(transfer_data.recipient, transfer_data.token_index, transfer_data.amount)`.
            - `new_pi.nonce === old_pi.nonce`.
            - Add `tx_hash` to `new_pi.nullifier_tree_root`.
            - Calculate `new_pi.block_hash` from `old_pi.block_hash`.
    - Deposit:
        - Private inputs:
            - Previous enough balance proof (140 KB).
            - Block validity proof (140 KB).
            - Deposit Merkle proof:
                - `index`.
                - `siblings` (320 bytes = 32 bytes * height 10).
            - Deposit data (68 bytes):
                - `recipient` (32 bytes).
                - `token index` (4 bytes).
                - `amount` (32 bytes).
                - Add transfer hash to the merged tx nullifier tree.
            - Fee transfer data:
                - `receiver` (32 bytes).
                - `token index` (4 bytes).
                - `amount` (32 bytes).
                - `salt` (8 bytes).
        - Constraints:
        **NOTE**: Denote old [new] public inputs as `old_pi` [`new_pi`].
            - `new_pi.public_key === old_pi.public_key`.
            - `recipient === old_pi.public_key`.
            - `new_pi.asset_tree_leaves[token_index] === old_pi.asset_tree_leaves[token_index] + deposit_data.amount`.
            - Hash the fee transfer data using Poseidon to get `fee_transfer_hash`.
            - `new_pi.asset_tree_leaves[fee_transfer_data.token_index] === old_pi.asset_tree_leaves[fee_transfer_data.token_index] - total_sent_amount`.
            - `new_pi.last_tx_hash === old_pi.last_tx_hash`.
            - `new_pi.last_block_number === old_pi.last_block_number`.
            - `deposit_hash = Hash(deposit_data.recipient, deposit_data.token_index, deposit_data.amount)`.
            - The deposit tree root of the block validity proof is `calcMerkleRoot(deposit_hash, deposit_merkle_proof.index, deposit_merkle_proof.siblings)`.
            - `new_pi.nonce === old_pi.nonce`.
            - Add `deposit_hash` to `new_pi.nullifier_tree_root`.
            - Calculate `new_pi.block_hash` from `old_pi.block_hash`.

**NOTE**: Users can set transfer amounts that exceed their balance. If an excessively large amount is sent, the account balance becomes negative. The public inputs of a balance proof with a negative balance do not record `last_tx_hash`, so the recipient cannot reflect this as a valid balance proof. The account can be reused by receiving an amount to offset the negative balance. Preventing accidental large transfers must be handled by the client side. Public inputs in the balance proof do not verify the authenticity of the public key. Hence, ownership of the public key is not a constraint.
